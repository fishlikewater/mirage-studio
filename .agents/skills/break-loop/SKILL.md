# 打破循环

用于修完 bug 后，防止“修一次、忘一次、再踩一次”。

## 1. 根因分类

| 类型 | 特征 |
|------|------|
| 缺少规范 | 不知道应该怎么做，靠猜 |
| 跨层契约不清 | 前端、Tauri、Rust、SQLite、模型协议理解不一致 |
| 传播遗漏 | 改了定义，漏改调用方、注册表、语言包或测试 |
| 验证缺口 | 单点验证通过，真实链路失败 |
| 隐含假设 | 代码依赖未写明的格式、时机、默认值或环境 |

## 2. 复盘问题

- 真正导致问题的最小原因是什么？
- 为什么第一次没有被类型检查、测试或 review 发现？
- 是否还有相同模式存在于节点、工具、模型、持久化或 i18n？
- 未来怎样让这个错误更难发生？

## 3. 立即沉淀

根据问题类型更新对应文档：

- 实现规则：`.trellis/spec/frontend/` 或 `.trellis/spec/backend/`
- 思考提醒：`.trellis/spec/guides/`
- 行为契约：对应 OpenSpec spec
- 项目专用能力：必要时更新 `.agents/skills/storyboard-provider-model-extension/`

不要只在聊天里写 TODO；如果值得记住，就落到文件里。

## 输出模板

```markdown
## Bug 复盘：<简述>

### 根因
- 类型：<缺少规范 / 跨层契约不清 / 传播遗漏 / 验证缺口 / 隐含假设>
- 具体原因：...

### 为什么会漏掉
- ...

### 防复发动作
- [ ] 更新 <spec 路径>
- [ ] 增加/调整验证 <命令或手测路径>
- [ ] 搜索并处理同类点 <关键字>
```

## 原则

调试的价值不只在修掉这个 bug，而在让同类 bug 之后更难出现。
