# 后端开发规范

## 默认基线

- 后端代码以清晰的业务边界、稳定的接口契约和可验证的行为为核心。
- 规范来源优先级：仓库已落地代码、配置、测试、迁移脚本、接口文档，然后才是本目录通用规则。
- 新增实现应贴合现有架构，不静默引入第二套分层、命名、错误模型或数据访问方式。
- 无法从现有代码判断约定时，采用本目录的保守默认规则，并在任务记录中说明假设。

## 使用原则

- 优先遵循仓库现有模式，不新增与现有风格冲突的写法。
- 先查已有公共模块、公共库或基础设施封装，再决定是否新增代码。
- Controller / API 层保持薄入口，业务规则放在服务、用例或领域层。
- 变更接口、数据结构、权限、异步链路或持久化模型时，同步更新调用方、测试和相关文档。
- 涉及跨模块、异步链路、权限边界时，同时阅读 `guides/` 下的思考指引。

---

## 默认完成标准

- 关键路径有自动化验证或明确的手工验证记录。
- 错误路径、空值、权限不足、重复提交等边界被显式考虑。
- 运行时配置、数据迁移、兼容性和回滚影响已说明。
- 不把“未运行测试”描述成“测试通过”。
