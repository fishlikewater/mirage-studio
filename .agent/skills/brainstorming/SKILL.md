# Brainstorming

Use this skill to turn an idea into a clear change request before code changes begin.

## Flow

1. Read only the context needed to understand the project area: `AGENTS.md`, `.cowork-flow/workflow.md`, and relevant spec indexes.
2. State assumptions, success criteria, and the smallest useful scope.
3. Ask one high-value question only when the answer cannot be inferred safely.
4. Present 2-3 viable approaches when trade-offs matter, with a concrete recommendation.
5. For L1/L2 work, create or update the change/task artifacts required by `.cowork-flow/workflow.md`.
6. Hand off to `writing-plans` when the requested behavior and acceptance criteria are clear.

## Output

Keep the result practical:

- Goal and non-goals.
- Chosen approach and rejected alternatives.
- Acceptance criteria.
- Open risks or blockers.

Do not implement during brainstorming unless the user explicitly changes the request to a direct edit.
