# 架构与边界规范

## 依赖与边界

- 模块间优先依赖接口和类型，不直接依赖具体实现细节。
- 跨模块通信优先使用明确的 service、port 或领域函数，不通过 UI 层绕路。
- 展示层不直接耦合基础设施层；Tauri/API 调用通过 `src/commands/**` 和应用层中转。

## 单一职责

- 一个文件只做一个业务概念；无法用三句话说清职责时，应评估拆分。
- 工具 UI、工具数据结构、工具执行逻辑分离：editor、annotation codec、processor 各守边界。
- Store 不承担重业务逻辑；业务规则放在 domain/application 层。

## 文件规模

- 舒适区：类或组件文件不超过 400 行，脚本不超过 300 行。
- 警戒线：800 行，必须评估拆分。
- 强制拆分：1000 行，纯数据定义除外。

## 层间通信

- 使用 DTO 或纯数据对象，避免双向引用。
- 检查数据流时按 `UI -> Store -> 应用服务 -> Tauri 命令/API -> 持久化` 追踪。
- 跨层改动必须同步错误表达、空态、加载态、回滚/失败路径和验证方式。

## 禁止模式

- UI 层手写节点类型白名单来绕过 registry。
- Store 中混入模型请求、图片处理、SQLite 语义等重流程。
- 为了“先跑通”新增与现有 registry、codec 或命令封装平行的一套映射。
